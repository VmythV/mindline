## 0. 文档说明与决策摘要

本文档基于与需求方的四轮澄清确认，核心决策已锁定如下：

| 决策项 | 结论 |
|--------|------|
| 产品定位 | **通用平台**：节点信息通过 JSON Schema 自定义，覆盖项目管理 / 知识管理 / OKR 等场景 |
| 平台形态 | **Web 网页端优先**；3D 走 WebGL/Three.js |
| 协作模式 | **实时协同编辑**（CRDT / Yjs） |
| 文档深度 | **PRD + 技术架构方案**（可交付开发动手） |
| AI 接入 | **多模型适配**（统一模型网关，可配置切换，自带默认） |
| 交付形态 | **SaaS 多租户 + 可私有化部署**（组件全部可自托管） |
| 规模 | **按可水平扩展设计**，先不过度优化 |
| 技术栈 | **React + TypeScript + Node**（与 Yjs、Three.js、React Flow 生态契合） |
| AI 拆解 | 四类能力全要（生成子树+预览 / 对话式 / 补全查重 / 总结改写），分期实现 |
| 权限 | **软隔离优先**：首版视图过滤 + 软权限；服务端硬隔离（内容子文档）放 M6 |
| 3D | **2D 编辑为主，3D 只读总览** |
| 里程碑 | **人工标记为主 + AI 辅助建议/摘要** |
| 渲染层 | **MVP 先用 React Flow 提速**，规模触顶后再评估自研渲染 |
| IM | **先手动发布**，订阅自动推送引擎放后期 |
| 落地节奏 | 认可 **M0 → M6** 分期路线图 |

## 1. 术语表（Glossary）

| 术语 | 含义 |
|------|------|
| Tenant（租户） | 一个企业/组织。SaaS 下多租户共存；私有部署下通常单租户 |
| Workspace（空间） | 租户下的工作空间，用于隔离不同团队/业务线（可选层级） |
| Project（项目） | 协作的基本单元，可父子嵌套；绑定一个思维导图 |
| MindMap（思维导图） | 一棵节点树，对应一个 Yjs 协同文档 |
| Node（节点） | 树上的一个信息单元，有类型、标题、自定义字段、负责人 |
| NodeTypeSchema（节点类型） | 定义某类节点有哪些字段及其类型/校验/UI 的 Schema |
| ChangeEvent（变更事件） | 一次语义级编辑的结构化记录，时间轴/历史的数据源 |
| Milestone（里程碑） | 时间轴上人工标记的重要节点，可附 AI 摘要 |
| CRDT | 无冲突复制数据类型，实时协同的数学基础（本项目用 Yjs） |
| Awareness | Yjs 的在线状态协议，承载光标、选区、在线用户 |
| 命令层（Command） | 封装一次用户编辑意图的对象，同时改 Yjs 文档并产出 ChangeEvent |
| 模型网关（AI Gateway） | 统一封装多家大模型的能力接口层 |
| 视图过滤 | 在有权范围内按负责人/状态等聚焦显示，隐藏项保留层级骨架 |
| 软权限 | 前端层面的内容隐藏（防君子）；区别于服务端硬隔离 |
| 硬隔离 | 服务端按权限裁剪下发，无权者根本拿不到内容（M6） |

## 2. 产品概述

### 2.1 背景与核心问题

团队在推进复杂工作（立项、需求拆解、目标分解、知识梳理）时，常面临三个痛点：

1. **从 0 到结构难**：面对一个大目标，不知如何拆成可执行的层级。
2. **过程不可追溯**：节点改来改去，最后没人说得清「为什么是现在这样」。
3. **协作割裂**：信息散落、权限混乱、人员变动后内容失管。

思谱 用「思维导图承载结构 + AI 降低拆解成本 + 时间轴沉淀过程」来系统性解决。

### 2.2 价值三角

- **思维导图**：结构化承载信息的骨架（2D 编辑为主，3D 总览）。
- **AI 拆解**：把「从模糊到结构」的成本降到最低，是差异化引擎。
- **时间轴**：把零散「变更」编织成可回顾的「叙事」，沉淀里程碑与决策。

### 2.3 目标用户

- 主要：中小企业 / 团队的项目负责人、产品、研发、运营、咨询顾问。
- 通用性：通过自定义节点类型，可被知识管理、OKR、需求管理等多场景复用。

### 2.4 范围

**在范围内（首版到完整版分期覆盖）**：节点树编辑、Schema 驱动自定义节点、AI 四能力、实时协同、变更记录与时间轴、人工里程碑+AI 辅助、视图过滤 + 权限、人员全局替换、父子项目、3D 总览、快捷键/右键/命令面板、IM 发布。

**明确不在范围内（首版）**：

- 甘特图 / 燃尽图等完整项目管理报表（时间轴是叙事轴，不是排期工具）。
- 审批流、工时统计、绩效。
- 移动端原生 App（Web 自适应；移动深度体验列为后续）。
- 完整离线优先（Offline-first）冲突合并（首版要求在线协同，仅容忍短暂断线重连）。

## 3. 核心概念与数据模型

### 3.1 概念关系

```
Tenant (租户/企业)
 └── Workspace (空间, 可选)
      └── Project (项目) ──┐ 可父子嵌套
           ├── 绑定一个 MindMap (Yjs 协同文档)
           ├── Members (成员 + 角色)
           └── NodeTypeSchemas (该项目可用的节点类型)

MindMap (Yjs 文档)
 └── Node (节点) ── 树结构 (parentId + 有序 children)
      ├── type            → 指向某 NodeTypeSchema
      ├── title           → Y.Text (协同标题, 轻富文本)
      ├── data            → 符合类型 Schema 的字段值 (Y.Map / Y.Text 混合)
      ├── ownerId         → 负责人
      ├── collaborators[] → 参与人
      ├── status / tags
      └── links[]         → 跨节点引用 / 下钻子项目

ChangeEvent (变更事件, 关系库)  ── who / when / nodeId / op / field / before / after
Milestone (里程碑, 关系库)       ── 人工标记 + 可选 AI 摘要
```

### 3.3 节点类型 Schema（自定义字段核心）

每个 `NodeTypeSchema.definition` 形态：

```jsonc
{
  "typeKey": "task",
  "displayName": "任务",
  "icon": "✅",
  "color": "#3B82F6",
  "fields": [
    { "key": "desc",     "label": "描述",     "type": "richtext", "collab": true },
    { "key": "priority", "label": "优先级",   "type": "enum", "options": ["高","中","低"], "default": "中" },
    { "key": "due",      "label": "截止日期", "type": "date" },
    { "key": "estimate", "label": "预估工时", "type": "number", "unit": "h" },
    { "key": "ac",       "label": "验收标准", "type": "richtext", "collab": true }
  ],
  "aiHints": "可执行任务节点；拆解时产出更细子任务，尽量填写优先级与验收标准。"
}
```

**支持的字段类型**：`text` / `richtext`（协同）/ `number` / `date` / `datetime` / `enum`（单选）/ `multiEnum`（多选）/ `user`（人员引用）/ `link`（URL）/ `checkbox` / `tags`。

**字段属性**：`key`、`label`、`type`、`required`、`default`、`options`（枚举）、`unit`、`collab`（是否协同富文本）、`uiHint`（控件提示）。

**协同存储约定**：`collab:true` 的字段存为 `Y.Text`（多人实时编辑）；其余结构字段存 `Y.Map`。

**内置开箱模板**（用户可改可扩，详见附录 A）：通用想法 `idea`、任务 `task`、目标 `objective` / 关键结果 `keyResult`、知识条目 `knowledge`、需求 `requirement`、缺陷 `bug`。

**Schema 变更兼容策略（A10，已确认）**：修改类型 Schema（删字段/改类型）后，默认**保留旧值并标记为废弃字段**（不破坏数据）；同时**提供 Schema 迁移工具**，可按规则**批量转换 / 重命名 / 清理**旧字段（例：字段改名、类型转换、枚举选项合并、删除废弃字段、为新必填字段批量填默认值）。迁移前可**预览影响范围**（涉及多少节点），迁移操作**记入变更历史**且支持回滚。Schema 带 `version`，节点记录所用版本以便溯源。

## 4. 功能需求

> 每个功能块给出：描述、关键交互、验收标准（AC）。优先级标注所属里程碑（M0–M6，见第 7 章）。

### F1 思维导图编辑（M0）

**描述**：以节点树为核心的 2D 编辑。

**交互**：
- 新建：`Tab` 建子节点，`Enter` 建同级兄弟节点。
- 删除：`Delete`（删除节点及子树，进回收/可撤销）。
- 移动：拖拽改父、改排序；快捷键上下移动。
- 折叠/展开：点击折叠柄或 `Cmd/Ctrl + .`。
- 节点正文：轻富文本（加粗/斜体/列表/链接/代码行内）。
- 自动布局（左右展开/径向）+ 手动微调位置。
- 视口虚拟化：仅渲染可视区 + 缓冲区节点。

**AC**：
- 两人同时编辑同一棵树，增删改移在 < 500ms 内互见且不丢操作。
- 删除子树后 `Cmd+Z` 可完整恢复结构与内容。
- 1000 节点的图滚动/缩放保持 ≥ 30 FPS（React Flow + 虚拟化）。

### F2 AI 拆解（四能力，M2 起分期）

**描述**：以选中节点为中心，结合关联上下文（父链 / 兄弟标题 / 子节点 / 该类型 Schema 与 aiHints）调用模型，产出结构化结果。

**能力 1 —— 生成子树（M2，首发）**：
- 选中节点 →「AI 拆解」→ 弹出选项（拆解层数、目标类型、补充提示词）。
- 模型按目标类型 Schema 产出结构化子节点（可多层）。
- **diff 预览面板**：以「将新增的子树」预览，逐个 **接受 / 拒绝 / 编辑**；可整体接受。
- 接受后经命令层落入树，记一条 `aiGenerate` 批次（`batch_id` 聚合）。

**能力 2 —— 对话式增量调整（M4）**：自然语言描述 →「在选中位置增/删/调整一片节点」，同样走预览确认。

**能力 3 —— 补全 / 查重（M4）**：扫描兄弟/子节点，提示「缺失项」「重复或可合并项」，给出操作建议。

**能力 4 —— 总结 / 改写（M2 摘要 / M4 改写）**：对子树生成摘要（供里程碑/IM 用）；对节点措辞改写润色。

**通用要求**：流式返回；可中断；失败/超时有重试与降级；模型输出强制 JSON 校验（见 6.4）。

**AC**：
- 生成子树结果 100% 经过预览，未确认不写入文档。
- 模型返回不合 Schema 时，系统自动重试 1 次并在仍失败时给出可读错误，不破坏现有树。
- 拆解上下文正确包含父链与兄弟标题（可在请求日志中验证）。

### F3 自定义字段 / Schema（M0 基础，M1 完善）

**描述**：节点字段由所属类型的 Schema 动态渲染；管理员可维护类型。

**交互**：
- 节点详情面板：按 Schema 渲染表单（enum→下拉、date→日期选择、richtext→协同编辑器、user→人员选择…）。
- Schema 编辑器：增删改字段、设默认值/必填/选项、调整顺序、设类型图标与色。
- 切换节点类型时按 A10 策略处理旧字段。
- **Schema 迁移工具（A10，排期 M4）**：字段定义变更时，可批量转换 / 重命名 / 清理旧字段；迁移前预览影响范围，迁移记入历史并支持回滚（详见 3.3）。

**AC**：新增一个字段后，所有该类型节点详情面板即时出现该字段且可编辑保存；执行一次字段重命名迁移后，所有受影响节点的旧字段值正确迁移到新字段且历史可查。

### F4 变更记录 & 时间轴（M1）

**描述**：所有编辑产生字段级可读历史；项目级时间轴叙事化呈现。

**交互**：
- 节点「历史」侧栏：按时间倒序列出字段级 diff（谁、何时、把 X 从 A 改成 B）。
- 项目**时间轴**：横向时间流，聚合 ChangeEvent；同 `batch_id`（如一次 AI 拆解）折叠为一条「批量事件」。
- 过滤：按人 / 操作类型 / 分支（子树）/ 时间范围。
- 时间轴上叠加里程碑标记（F5）。

**AC**：
- 任一字段修改都能在历史中查到操作人、时间、前后值。
- 一次生成 20 个子节点的 AI 拆解，在时间轴上折叠为 1 条可展开的批量事件。

### F5 里程碑（人工为主 + AI 辅助，M2）

**描述**：人工在时间轴/节点标记里程碑；AI 辅助识别与摘要。

**交互**：
- 手动「标记里程碑」：填名称 + 说明，可锚定节点、关联时间区间。
- AI 辅助①：扫描近期变更，**建议**「这段时间发生了 X，可能是里程碑」（建议态，用户采纳才生成）。
- AI 辅助②：为里程碑关联区间自动生成「阶段大事件摘要」初稿，人工可改。

**AC**：里程碑可在时间轴上显示并点击展开；AI 摘要为可编辑初稿而非锁定文本。

### F6 权限 & 视图过滤（软隔离优先，M3）

**描述**：分两层——「能不能看」=权限层；「想不想看」=视图过滤层。首版以视图过滤 + 软权限为主。

**权限层**：
- 项目角色：Owner / Admin / Editor / Commenter / Viewer。
- 软权限：节点/分支可标「私有」，前端对无权者隐藏内容（防君子，不防小人）。

**视图过滤层（首版重点，贴合需求方描述）**：
- 按 **负责人 / 状态 / 标签 / 「我的」** 过滤聚焦。
- 过滤后**未命中节点折叠为骨架灰条**（标题脱敏或仅占位），**保留逐级层级连线** —— 即「单独看 A 的内容时仍能看到一级一级的层级结构」。
- 可一键在「看全部 / 只看某人」之间切换。

**硬隔离（M6，非首版）**：节点正文拆「内容子文档」按需鉴权下发，无权者只同步骨架，从根上拿不到内容。

**AC**：
- 选择「只看 A」后，画布仅高亮 A 负责的节点内容，其余以灰条骨架显示且层级连线完整。
- 切回「看全部」恢复完整内容。

### F7 人员全局替换（离职移交，M3）

**描述**：把某用户在所有引用处批量替换为另一人。

**交互**：
- 选「源人员 → 目标人员」。
- 系统扫描所有 `ownerId` / `collaborators` / `@提及` 引用 → **预览影响范围**（涉及 N 个节点、M 个项目）。
- 确认执行 → 批量替换 + 记一条 `transfer` 变更 + 可选 IM 通知接收人。
- **历史变更的「原作者」保留不变**（保真），只替换「当前负责/分配」。

**AC**：替换后源用户不再出现在任何「当前负责人/参与人」位置；历史记录中原操作人不被篡改；操作本身可在时间轴查到。

### F8 多人实时协作（M0 内核 / M1 体验）

**描述**：基于 Yjs 的实时协同。

**交互**：
- 实时同步增删改移与字段编辑。
- Awareness：在线用户头像、他人光标/选区、节点「正在编辑」提示。
- 评论 & @：节点级评论，@ 触发通知（M4 起可联动 IM）。
- 撤销重做：`UndoManager` 仅撤销**自己**的操作（A9，多人下不误撤他人改动）。

**AC**：两端在线编辑互见延迟 < 500ms；断网 10 秒重连后自动合并无冲突丢失；撤销不影响他人操作。

### F9 父子项目 & 跨项目引用（M4）

**描述**：项目可嵌套；节点可下钻子项目；支持跨项目引用。

**交互**：
- 项目树：项目可有父子层级。
- 节点「下钻」：某节点链接到一个子项目/子导图（A7：子项目是**独立 MindMap**，非内联展开）。
- 跨项目引用：节点可引用另一项目节点（镜像链接，点击跳转，不复制内容）。

**AC**：从父项目节点可跳转到其子项目导图；跨项目引用在源节点变更后跳转目标始终最新。

### F10 3D 总览（只读，M5）

**描述**：react-three-fiber 把树渲染为 3D 层级总览，用于鸟瞰/演示/汇报。

**交互**：
- 3D 布局（径向球面 / 分层悬浮）；漫游、旋转、缩放。
- 只读：点击节点**下钻定位回 2D 编辑**，不在 3D 内直接编辑。
- 性能：实例化渲染 + LOD + 视锥剔除，扛大节点量。

**AC**：5000 节点的图在 3D 总览下可流畅旋转浏览（≥ 30 FPS）；点击节点能定位回 2D 对应节点。

### F11 快捷键 / 右键 / 命令面板 / 预览（M1）

**描述**：高效操作体系（完整清单见附录 B/C）。

**要点**：
- 快捷键：`Tab`/`Enter`/`Delete`/`Cmd+Z`/`Cmd+Shift+Z`/方向键导航、`/` 召唤 AI、`Space` 预览节点详情（不进编辑）、`Cmd+K` 命令面板。
- 右键菜单：上下文操作（AI 拆解、设负责人、标里程碑、复制链接、发布到 IM、下钻子项目…）。
- 节点预览：`Space` 或悬停弹出详情卡片，只读速览。

**AC**：所有高频操作均有键盘路径；命令面板可模糊搜索并执行主要命令。

### F12 IM 发布（先手动，M4；订阅推送后期）

**描述**：把节点/里程碑/摘要发布到 IM；渠道适配多家。

**交互**：
- 渠道：企业微信 / 钉钉 / 飞书 / Slack / 通用 Webhook（适配层抽象）。
- **手动发布（首发）**：右键/菜单「发布到 IM」→ 选渠道与群 → 生成消息卡片（标题 + 摘要 + 链接）发送。
- **订阅自动推送（后期）**：规则触发（分支变更 / 里程碑达成 / @我）自动推送，需订阅引擎，排期靠后。

**AC**：手动发布能把选中节点/里程碑作为卡片消息送达目标 IM 群，消息含可点击回链。

### 5.3 实时协同设计（Yjs / Hocuspocus）

**协同文档结构（每个 MindMap 一个 Y.Doc）**：

```
Y.Doc
 ├── nodes: Y.Map<nodeId, Y.Map>      // 所有节点的扁平索引
 │     └── 每个节点 Y.Map:
 │           ├── parentId: string
 │           ├── order: number/fracIndex  // 同级排序(分数索引避免重排冲突)
 │           ├── type: string
 │           ├── title: Y.Text           // 协同标题
 │           ├── ownerId / status / tags
 │           ├── collaborators: Y.Array
 │           ├── data: Y.Map             // 结构字段
 │           │     └── richtext字段: Y.Text
 │           └── links: Y.Array
 └── meta: Y.Map                         // 文档级元信息
```

- **排序**用分数索引（fractional indexing），插入/移动不需重排整组，减少冲突。
- **撤销重做**：`Y.UndoManager` 绑定本地 origin，仅撤销本人操作（A9）。
- **在线协作**：Awareness 承载光标位置（节点 id + 字段 + 选区）、用户头像、颜色。
- **持久化**：Hocuspocus `onStoreDocument` 周期把 Y.Doc 编码为二进制 update 落 Postgres；定期快照压缩历史。
- **多实例扩展**：Redis pub/sub 广播 update，支持协同服务水平扩展。
- **鉴权**：`onAuthenticate` 校验 token + 项目成员资格 + 文档读写权限。

### 5.4 命令层 & 变更事件派生（D7）

- 所有用户编辑封装为**命令对象**（如 `CreateNode` / `MoveNode` / `SetField` / `SetOwner` / `DeleteSubtree`）。
- 命令执行：① 在 Yjs 事务内修改文档（带 origin 标记）；② 同步产出结构化 `ChangeEvent`。
- ChangeEvent 经 API 落 `change_events` 表（或经 Hocuspocus `onChange` 二次校验）。
- 批量操作（AI 拆解、人员替换）共享一个 `batch_id`，时间轴可折叠。
- **好处**：时间轴/历史基于语义事件（可读、可 diff），而非碎片化的 CRDT update。

### 5.5 AI 模型网关（D6）

- 对上提供四个能力接口：`decompose`（生成子树）、`complete`（补全查重）、`summarize`（总结）、`rewrite`（改写）。
- 对下适配多家：**OpenAI 兼容协议优先**（国产模型多数兼容），各家 SDK 作补充；私有部署可接自托管模型（vLLM 等）或客户自带 Key。
- **结构化输出**：用 JSON mode / function calling 约束输出为符合目标类型 Schema 的结构；服务端做 JSON Schema 校验，失败重试 1 次再降级。
- **上下文组装**：父链标题 + 兄弟标题 + 现有子节点 + 目标类型 Schema/aiHints + 用户补充提示。
- **流式**：SSE/WebSocket 向前端流式返回，可中断。
- **成本（A6）**：SaaS 提供默认额度 + 企业可配自带 Key；按租户/项目计量。

### 5.6 权限模型实现

- **首版（软隔离）**：权限校验在 API 与协同 `onAuthenticate`（文档读写级）；节点/字段「私有」由前端按成员角色过滤渲染。视图过滤纯前端，不涉及同步边界。
- **进阶（硬隔离，M6）**：把节点 `data` 中敏感正文拆为**内容子文档**（独立 Y.Doc 或子 map），骨架文档（结构 + 标题占位）与内容文档分离同步；无权者只连骨架文档，内容文档经鉴权按需加载。

### 5.7 多租户 & 私有部署（D10）

- **多租户**：所有业务表带 `tenant_id`，行级隔离 + 应用层强制租户上下文；协同文档与对象存储按 tenant 命名空间隔离。
- **私有部署**：同一套镜像；依赖（Postgres/Redis/MinIO/Hocuspocus/模型网关）全部可自托管；一键 Compose / K8s Helm。私有环境无外网时模型网关指向本地自托管模型或客户 Key。

## 8. 风险与权衡

1. **大规模 + Yjs 单文档**：上万节点时整文档同步/内存压力大。缓解：内容子文档拆分、子树懒加载、正文延迟同步。
2. **权限硬隔离与 CRDT 矛盾**：彻底隔离需重构同步边界，故放 M6；首版软隔离对常规场景足够。
3. **渲染层选型**：React Flow 提速但大规模/深度定制受限；预留自研 Canvas 替换路径（命令层与数据模型与渲染解耦，便于替换）。
4. **AI 结构化输出不稳定**：JSON mode + 服务端校验 + 重试 + 人工 diff 兜底。
5. **多模型能力差异**：网关做能力探测与降级，统一 OpenAI 兼容协议优先。
6. **私有部署模型可用性**：支持自托管开源模型或客户自带 Key。
