## 1. 两层模型（核心区分）

| 层 | 解决 | 性质 | 首版 |
|----|------|------|------|
| **权限层** | 「能不能看/改」 | 强约束（角色 + 私有标记） | 软隔离（前端过滤 + 连接级鉴权） |
| **视图过滤层** | 「想不想看（聚焦谁）」 | 弱约束（纯展示） | 完整支持 |

> 需求方最新描述（「能看全部，也能单独看 A 的内容，且保留层级」）主要属于**视图过滤层**；原始需求「只看自己内容」属于**权限层**。两者都做，互不耦合。

## 2. 角色权限矩阵（项目级）

| 能力 \ 角色 | Owner | Admin | Editor | Commenter | Viewer |
|-------------|:-----:|:-----:|:------:|:---------:|:------:|
| 查看项目/树 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 编辑节点结构/内容 | ✓ | ✓ | ✓ | ✗ | ✗ |
| 评论/@ | ✓ | ✓ | ✓ | ✓ | ✗ |
| AI 拆解/写入 | ✓ | ✓ | ✓ | ✗ | ✗ |
| 管理成员/角色 | ✓ | ✓ | ✗ | ✗ | ✗ |
| 编辑节点类型 Schema | ✓ | ✓ | ✗ | ✗ | ✗ |
| 执行 Schema 迁移 | ✓ | ✓ | ✗ | ✗ | ✗ |
| 人员全局替换 | ✓ | ✓(限本项目) | ✗ | ✗ | ✗ |
| 配置 IM 渠道/发布 | ✓ | ✓ | 发布✓/配置✗ | ✗ | ✗ |
| 删除/归档项目 | ✓ | ✗ | ✗ | ✗ | ✗ |

- 角色定义于 `project_members.role`。
- 租户级超管（可选）：跨项目管理，用于 SaaS 运营/私有部署管理员。
- 父子项目：子项目可**继承**父项目成员与角色（可开关）；也可单独授权。

## 3. 软权限（节点/分支私有，首版）

### 3.1 规则

- 节点字段 `private: boolean`（存于 Y.Doc，见 Yjs 详设 §2）。
- 私有可作用于**单节点**或**整条分支**（设分支根 private，子孙继承，除非子节点显式 public）。
- **可见判定**（前端）：
  ```
  canSeeContent(node, user) =
      !node.effectivePrivate
   || user 是 node.owner / collaborators
   || user.role ∈ {Owner, Admin}
  ```
- `effectivePrivate` = 自身 private 或最近祖先 private（继承）。

### 3.2 不可见时的呈现（贴合「保留层级」）

无权看内容的节点**不消失**，而是渲染为**骨架灰条**：
- 显示：层级位置、连线、占位标题（如「🔒 受限节点」或脱敏标题）。
- 隐藏：正文、自定义字段值、负责人（可配是否露出负责人）。
- 仍可展开/折叠以浏览结构。

> 这正是「单独看某人内容时仍能看到一级一级层级结构」的呈现基础——结构始终可见，内容按权限/过滤决定。

### 3.3 软隔离的边界（诚实声明）

首版**服务端仍全量下发**文档（Yjs 整文档同步），私有内容仅前端隐藏 → **防君子不防小人**（懂技术者可从同步数据读到）。适合内部协作、低敏感场景。强合规场景需 §5 硬隔离。

## 4. 视图过滤层（首版重点，纯展示）

### 4.1 过滤维度

- 按**负责人**：「只看 A」「只看我的」「多选若干人」。
- 按**状态** / **标签** / **类型** / **时间范围**。
- 组合过滤（AND / OR 可选）。

### 4.2 过滤行为

```
对每个节点计算 match(node, filter):
  - 命中 → 正常显示
  - 未命中 → 折叠为骨架灰条（同 §3.2 呈现），但：
      * 若其子孙有命中 → 保留为「路径节点」（半显，确保层级连贯）
      * 若整条分支无命中 → 可整体折叠为一个「N 个隐藏节点」摘要块
```

- **关键**：过滤永远**保留从根到命中节点的层级骨架**，不破坏树结构认知。
- 一键切换：`看全部 ⇄ 只看某人`，状态存于本地视图设置（不影响他人）。
- 与软权限叠加：先按权限决定「能否看内容」，再按过滤决定「显不显示内容」；二者都是「显示/隐藏」，呈现一致（骨架灰条）。

### 4.3 与协同关系

视图过滤是**纯本地 UI 状态**，不进 Y.Doc、不影响他人视图、不影响同步。

## 5. 硬隔离演进（M6，非首版）

当需要「无权者根本拿不到内容」时：

### 5.1 内容子文档拆分

```
骨架文档 (skeleton Y.Doc, 每 map 一个)
   nodes: { id, parentId, order, type, ownerId?, private, titleMasked }
   （只含结构 + 占位标题，人人可同步）

内容文档 (content Y.Doc, 按节点或按分支分片)
   { title 全文, data 全字段, richtext... }
   （按权限鉴权后才下发对应分片）
```

### 5.2 同步策略

- 所有成员同步**骨架文档** → 人人可见层级结构（满足「体现层级」）。
- 内容文档分片：客户端按需请求 → Hocuspocus `onAuthenticate`/路由层按 `canSeeContent` 决定是否下发该分片。
- 无权分片：服务端根本不发送 → 真正隔离。

### 5.3 代价与权衡

- 复杂度高：双文档一致性、分片粒度（按节点太碎/按分支太粗）、跨分片操作（移动节点跨权限域）。
- 故放 M6；首版数据模型已为此预留（`data` 与骨架字段分离的结构便于后续拆分）。

## 6. 人员全局替换（F7，M3）

### 6.1 引用面清单

替换需扫描所有「**当前**指向某用户」的位置：
- `node.ownerId`
- `node.collaborators[]`
- 节点正文/评论中的 `@提及`（mention）
- 项目成员 `project_members`（视选项：移交是否含成员席位）
- 订阅/通知接收人（IM 订阅，若已有）

**不替换**：`change_events.actor_id`、`milestones.created_by` 等**历史归属**（保真，谁干的就是谁干的）。

### 6.2 流程

```
选择 from_user → to_user，scope（本项目 / 工作空间 / 全租户）
  │
  ├─ preview： 扫描引用 → 返回影响范围
  │     { projects: n, nodes: m, mentions: k, memberships: j }
  │     （列出明细供人工核对，可排除个别项）
  │
  └─ execute：
        后台任务分批替换（大范围异步）
        每个被改节点经命令层产出 ChangeEvent(op=transfer, batchId)
        生成一条「移交记录」（from/to/操作人/时间/范围）
        可选：IM 通知 to_user「你接收了 X 的 m 个节点」
```

### 6.3 一致性与并发

- 执行期对涉及文档加「移交进行中」轻量标记（避免与并发编辑互相覆盖关键字段）。
- 失败可重入（幂等：按「仍等于 from_user 才改为 to_user」判定）。
- `from_user` 状态可顺带置 `left`（离职）。

### 6.4 验收

- 替换后 `from_user` 不再出现在任何「当前负责人/参与人/@」位置。
- 历史记录中 `from_user` 作为原操作人**不变**。
- 移交操作本身在时间轴可查（transfer 批量事件）。

## 7. 权限校验落点汇总

| 落点 | 校验内容 |
|------|----------|
| REST API（NestJS Guard） | JWT + 租户上下文 + 项目角色 + 操作权限 |
| 协同连接（onAuthenticate） | 成员资格 + map 读/写 |
| 命令层 | 写操作前校验角色（Editor+）与节点私有约束 |
| 前端渲染 | 软权限内容隐藏 + 视图过滤（展示层） |
| 硬隔离（M6） | 内容子文档分片下发鉴权 |

## 8. 待定/风险

- 软隔离「防君子不防小人」需向客户**明确说明**，避免误期望；高敏感客户引导至 M6 或私有部署。
- 视图过滤「路径节点半显」的具体视觉规则需 UX 细化（半显 vs 仅连线）。
- 人员替换的 scope=全租户在大数据量下是重后台任务，需进度反馈与可取消。
- 子项目权限继承的开关语义（继承后再单独改的优先级）需在实现时定清。

## 9. 待定点敲定（原 §8）

| 原待定 | 敲定结论 |
|--------|----------|
| 软隔离须明确声明 | 项目设置页显著标注「软隔离：私有内容仅前端隐藏，技术上仍随文档下发；强隔离请用私有部署或等待 M6 硬隔离」；用户首次设节点私有时弹确认。 |
| 路径节点半显规则 | 未命中但有命中子孙的祖先 → 半透明，仅标题 + 连线，不显内容/字段；整条无命中分支 → 折叠为「N 个隐藏节点」摘要块，可点开临时展开。 |
| 子项目权限继承 | 默认 `inheritMembers=true`（继承父成员与角色）；子项目可追加成员或对个别成员覆盖角色；优先级「子项目显式设置 > 继承」；覆盖不得超过该用户在租户的权限上限。 |

## 11. 接口契约

### GET /api/projects/:id/permissions
返回当前用户在该项目的能力集（前端据此控制 UI 显隐）。
```jsonc
// 响应 200
{ "role":"editor",
  "can": { "edit":true,"comment":true,"aiWrite":true,"manageMembers":false,
           "editSchema":false,"runMigration":false,"transfer":false,"publishIM":true } }
```

### PATCH /api/projects/:id/members/:userId
```jsonc
// 请求 { "role": "admin" }   // 仅 Owner/Admin；不可超租户上限
// 响应 200 { "userId":"u_2","role":"admin" }
```

### POST /api/transfer/preview
```jsonc
// 请求
{ "fromUserId":"u_5","toUserId":"u_9","scope":"project|workspace|tenant","scopeId":"p_1" }
// 响应 200
{ "impact": { "projects":3,"nodes":126,"mentions":14,"memberships":3 },
  "details":[ {"projectId":"p_1","nodes":80,"sampleNodeIds":["n_1","n_2"]} ] }
```

### POST /api/transfer/execute
```jsonc
// 请求 (同 preview + 可选 exclude)
{ "fromUserId":"u_5","toUserId":"u_9","scope":"tenant","exclude":{"nodeIds":["n_9"]} }
// 响应 202 { "jobId":"job_x","status":"running" }
```

### GET /api/transfer/:jobId
```jsonc
// 响应 200
{ "jobId":"job_x","status":"running|done|failed","processed":80,"total":126,
  "conflicts":[ {"nodeId":"n_30","reason":"当前值已非 from"} ] }
```
错误：403 无 transfer 权限 · 409 已有进行中的同范围任务。
